## 🔴 Triệu chứng Cực kỳ Nguy hiểm (Critical Issues) - Cần Phẫu Thuật Ngay!

### 1. Triệu chứng Rò rỉ Credentials qua đường dẫn UNC (NTLM Hash Leak)
- **Vị trí phát hiện:** File [app.go](file:///d:/skul9x/dev/MusicYT-main/app.go#L60-L88) - Hàm `OpenOutputFolder(folderPath string)`
- **Triệu chứng bệnh:** Hàm này nhận vào một `folderPath` từ frontend, kiểm tra sự tồn tại bằng `os.Stat(folderPath)` và mở nó bằng `exec.Command("explorer", folderPath)` trên Windows.
- **Mức độ nguy hiểm:** **Cực kỳ nguy hiểm (Hacker có thể đánh cắp mật khẩu Windows của người dùng)**.
- **Nếu không sửa, chuyện gì sẽ xảy ra?** Nếu có lỗ hổng XSS ở frontend hoặc người dùng bị lừa dán một đường dẫn mạng dạng UNC (ví dụ: `\\attacker-server\share`), hàm `os.Stat` và lệnh `explorer` trên Windows sẽ tự động thực hiện kết nối SMB đến máy chủ của kẻ tấn công. Windows sẽ tự động gửi mã băm mật khẩu của người dùng hiện tại (NTLM Hash) sang máy chủ đó để xác thực. Kẻ tấn công chỉ việc thu thập mã băm này và dùng các công cụ bẻ khóa để lấy mật khẩu Windows của người dùng trong vài giây.
- **Phác đồ điều trị (Cách sửa):**
  Thêm bước kiểm tra chặn tuyệt đối các đường dẫn bắt đầu bằng ký tự dạng UNC (`\\` hoặc `//` hoặc các giao thức mạng) trước khi thực hiện `os.Stat` và gọi explorer.
  ```go
  // Chặn đường dẫn UNC mạng để tránh rò rỉ NTLM Hash
  cleanPath := filepath.Clean(folderPath)
  if strings.HasPrefix(cleanPath, `\\`) || strings.HasPrefix(cleanPath, `//`) {
      return fmt.Errorf("không hỗ trợ đường dẫn mạng UNC để đảm bảo bảo mật")
  }
  ```

### 2. Triệu chứng "Liệt Vận Động" sau khi cài đặt Dependency (PATH Propagation Bug)
- **Vị trí phát hiện:** File [app.go](file:///d:/skul9x/dev/MusicYT-main/app.go#L167-L212) - Hàm `SetupWindowsDependencies()` phối hợp với `CheckDependencies()`
- **Triệu chứng bệnh:** Khi người dùng bấm cài đặt tự động trên Windows, ứng dụng sẽ tải `yt-dlp.exe` và `ffmpeg` về thư mục `C:\tools`, sau đó chạy Powershell để sửa biến môi trường `PATH` của User. Tuy nhiên, biến môi trường PATH của tiến trình hiện tại (Wails app đang chạy) **không hề thay đổi** vì tiến trình con không thể ghi đè biến môi trường của tiến trình cha. Do đó, hàm `CheckDependencies` vẫn báo thiếu công cụ cho đến khi người dùng tắt hẳn app đi và bật lại.
- **Mức độ nguy hiểm:** **Gây khó chịu cực độ cho trải nghiệm người dùng (UX Break/Bug)**. Người dùng sẽ tưởng công cụ bị lỗi vì bấm "Cài đặt thành công" rồi mà app vẫn cứ báo thiếu và bắt cài lại.
- **Phác đồ điều trị (Cách sửa):**
  Trong hàm `CheckDependencies()`, ngoài việc dùng `exec.LookPath`, chúng ta hãy chủ động kiểm tra xem `C:\tools` có tồn tại và chứa file không. Nếu có, hãy tự động nạp `C:\tools` vào biến môi trường của tiến trình hiện tại ngay lập tức bằng code Go:
  ```go
  // Trước khi check, nếu trên Windows ta chủ động add C:\tools vào PATH của tiến trình hiện tại
  if runtime.GOOS == "windows" {
      toolsDir := "C:\\tools"
      if _, err := os.Stat(filepath.Join(toolsDir, "yt-dlp.exe")); err == nil {
          currentPath := os.Getenv("PATH")
          if !strings.Contains(currentPath, toolsDir) {
              os.Setenv("PATH", currentPath+";"+toolsDir)
          }
      }
  }
  ```

## 🟡 Cảnh báo Quan trọng (Warnings) - Nên Điều Trị Sớm

### 3. Tải file không có cơ chế kiểm tra tính toàn vẹn (No Hash Validation)
- **Vị trí phát hiện:** File [app.go](file:///d:/skul9x/dev/MusicYT-main/app.go#L167-L212) - Hàm `SetupWindowsDependencies()`
- **Triệu chứng bệnh:** Tải trực tiếp `yt-dlp.exe` từ GitHub qua URL mà không kiểm tra chữ ký hay mã băm SHA256.
- **Nếu không sửa, chuyện gì sẽ xảy ra?** Nếu người dùng sử dụng mạng công cộng bị hack hoặc DNS bị giả mạo (DNS Poisoning/MITM), ứng dụng có thể tải về một file `yt-dlp.exe` giả mạo chứa virus/trojan và chạy nó trực tiếp trên hệ thống của người dùng.
- **Phác đồ điều trị (Cách sửa):** Dù tải từ GitHub là phổ biến, nhưng tốt nhất nên dùng giao thức HTTPS chuẩn (đã có), và tối ưu nhất là hiển thị cảnh báo cho người dùng hoặc có cơ chế check size/version cơ bản.

### 4. Triệu chứng "Mồ Côi Tiến Trình" do thiếu nút Hủy Tải (Orphaned Processes & No Cancellation)
- **Vị trí phát hiện:** File [app.go](file:///d:/skul9x/dev/MusicYT-main/app.go#L265-L320) - Hàm `DownloadVideo()` và `DownloadGenericVideo()`
- **Triệu chứng bệnh:** Tiến trình `yt-dlp` được kích hoạt và chạy độc lập. Không có cơ chế truyền `context.Context` có khả năng hủy (`WithCancel`) để dừng tiến trình khi người dùng tắt ứng dụng hoặc muốn dừng tải giữa chừng.
- **Nếu không sửa, chuyện gì sẽ xảy ra?** Khi một file nhạc cực lớn đang được tải, nếu người dùng tắt app hoặc chuyển tab, tiến trình `yt-dlp` vẫn âm thầm chạy ngầm trong Windows Task Manager, gây ngốn băng thông internet và CPU một cách lãng phí mà không báo trước.
- **Phác đồ điều trị (Cách sửa):** 
  Sử dụng `exec.CommandContext(ctx, ...)` thay cho `exec.Command(...)`, lưu trữ con trỏ hủy tiến trình (cancel function) trong App struct để cho phép người dùng nhấn nút "Hủy" trên giao diện React.

## 🟢 Đề xuất Nâng cao (Suggestions) - Giúp App Trẻ Khỏe Hơn

### 5. Rủi ro xung đột tiến trình tải đồng thời (Race Condition)
- **Triệu chứng bệnh:** Biến trạng thái `progress` và `isDownloading` dùng chung ở file `App.tsx` cho cả hai Tab. 
- **Giải pháp:** Mặc dù nút tab đã bị khóa (`disabled={isDownloading}`) khi đang tải để ngăn xung đột, nhưng về lâu dài, nếu muốn hỗ trợ tải nhiều file cùng lúc, anh nên cấu trúc danh sách tiến trình tải ở Frontend theo dạng mảng/map các ID tiến trình riêng biệt thay vì một biến đơn lẻ.

### 6. Cần bổ sung Logging tập trung
- **Triệu chứng bệnh:** Các lỗi chỉ được trả về thô sơ qua `error` hoặc in ra console bằng `console.error`.
- **Giải pháp:** Nên lưu file log chẩn đoán tại thư mục `.brain` hoặc thư mục tạm của hệ thống để dễ dàng debug khi người dùng báo lỗi tải thất bại.
